# Tachyon Core Alpha VPS Test Plan

This plan turns a real VPS relay test into a repeatable checklist. Tachyon Core
is still an alpha component, not a stable or production-complete release.

## Scope

Use this plan to validate a Tachyon Core server relay on a real VPS and to
collect shareable diagnostics for the client alpha loop. It covers server setup,
safe relay ACL selection, local smoke tests, VPS checks, and the redacted output
to send back.

Do not use this plan to validate client TUN takeover, system proxy behavior, or
full production readiness.

## VPS Prerequisites

- Debian or Ubuntu VPS with shell access and `sudo`.
- Inbound UDP opened in both the host firewall and cloud security group for the
  Tachyon server listen port, for example UDP `443` or another explicit port.
- Outbound UDP from the VPS to the selected game server targets.
- A domain name and certificate only if your chosen deployment profile requires
  them. The basic TGP relay path uses UDP plus PSK and does not require a TLS
  certificate by itself.
- A private `tgp.auth.psk` generated by the installer or by a secure secret
  generator. Treat it like a password. Never paste it in issues, chat, logs, or
  screenshots.
- A small, explicit list of game UDP destinations for
  `server.relay.allowed_targets`.

## Choose `allowed_targets`

`server.relay.allowed_targets` is the relay allow-list. It must describe the
real game UDP targets that this VPS may forward to.

Good examples:

```text
cidr=198.51.100.0/24,ports=27015-27050
domain=game.example.com,ports=27015
domain=region1.game.example.com,ports=7000,7001,7002
```

Selection rules:

- Prefer the narrowest documented game server CIDR or domain for the region you
  are testing.
- Always include explicit UDP ports or port ranges.
- Keep separate rules for separate games or regions so failures are easy to
  diagnose.
- If you do not know the target yet, leave `allowed_targets` empty and accept
  the safe deny-all state until you can fill it in.

Forbidden:

- Wildcards such as `0.0.0.0/0` or `::/0`.
- Any open relay shape that can forward to the whole internet.
- Missing `ports`.
- Copying subscription proxy hosts into `allowed_targets` unless they are
  actually the game UDP destination.

## Local Smoke Before VPS

From the `tachyon-core` repository, run:

```bash
bash scripts/smoke-tgp-relay.sh
```

This is a local-only smoke test. It binds temporary `127.0.0.1` UDP ports and
checks PSK handshake behavior, missing/wrong PSK rejection, config-driven
client/server relay wiring, ACL allow/deny, deny-all defaults, wildcard target
rejection, and an echo-like UDP relay round trip.

It does not use your VPS, cloud security group, real client, real game server,
TUN, Prism, Xray, routes, firewall, systemd, Docker, or system proxy settings.
Passing local smoke means the relay logic and config-to-runtime mapping are
sane; it does not prove VPS reachability.

## Bare-Metal Deployment Path

Run on the VPS:

```bash
sudo bash scripts/install-server.sh --port 443 \
  --allow-target 'cidr=198.51.100.0/24,ports=27015-27050'
```

For reproducible alpha testing, pass the intended release tag instead of relying
on the moving latest entry:

```bash
sudo bash scripts/install-server.sh --version v0.1.0-alpha.15 --port 443 \
  --allow-target 'domain=game.example.com,ports=27015'
```

After installation:

1. Confirm the VPS firewall and cloud security group allow inbound UDP on the
   configured listen port.
2. Copy the generated `tgp.auth.psk` value from the server config into the Prism
   Tachyon server profile.
3. Do not send the PSK back to the project.

## Docker Deployment Path

Run on the VPS:

```bash
sudo TACHYON_ALLOWED_TARGETS='domain=game.example.com,ports=27015' \
  bash scripts/install-server-docker.sh --version v0.1.0-alpha.15 --port 443
```

The Docker path mounts the downloaded static `tachyon-core` binary into a
`debian:bookworm-slim` container. It does not require a GHCR image.

After installation, check both the host UDP exposure and the container state.
The cloud security group must still allow inbound UDP to the published server
port.

## Post-Deployment Verification

Use the read-only verifier that matches the deployment type:

```bash
sudo bash scripts/verify-server.sh
sudo bash scripts/verify-server.sh --mode docker
```

For a copied binary/config pair before service start:

```bash
bash scripts/verify-server.sh --mode config --binary ./tachyon-core --config ./server.json
```

`verify-server.sh` validates the binary and config, checks service or container
state when requested, summarizes `allowed_targets`, inspects UDP listener state,
and tails logs with PSK redaction. It does not change firewall rules, cloud
security groups, Docker, systemd, packet filters, routes, or proxy settings.

## Support Bundle

When asking for help, prefer a read-only support bundle instead of manually
copying several command outputs:

```bash
sudo bash scripts/collect-server-diagnostics.sh
sudo bash scripts/collect-server-diagnostics.sh --mode docker
```

The collector creates a timestamped `tachyon-server-diagnostics-*.tar.gz` in
the current directory. If chat or email cannot accept archives, generate a
single text report instead:

```bash
sudo bash scripts/collect-server-diagnostics.sh --format txt
```

The bundle includes OS/kernel details, Tachyon Core version, config validation
summary, `allowed_targets` summary, service/container status, UDP listener
state, redacted journal or Docker log tails, and a redacted `verify-server.sh`
summary/output. It is intended to be read-only and does not start, stop, reload,
or reconfigure systemd, Docker, firewall, iptables, nftables, routes, or proxy
settings.

The collector redacts common PSK, token, UUID, private key, password, and
subscription/proxy URL forms, but you must still manually inspect the generated
file before sending it back.

## VPS Smoke vs Local Smoke

- Local smoke: `scripts/smoke-tgp-relay.sh`; runs only on loopback and proves
  relay behavior plus config-driven runtime wiring without host networking
  changes.
- VPS verification: `scripts/verify-server.sh`; runs on the deployed server and
  proves the installed binary/config/listener shape.
- Real client smoke: Prism connects to the VPS Tachyon server profile and
  validates client-side config, launch, logs, and game-mode/manual rules.
- Real game smoke: a real game emits UDP that matches the selected profile and
  `allowed_targets`; this is the field result that local smoke cannot replace.

## Output to Send Back

Send:

- VPS OS and version, for example Debian 12 or Ubuntu 24.04.
- Deployment path: bare metal or Docker.
- Exact install command with PSK removed.
- Exact verify command.
- The generated `tachyon-server-diagnostics-*.tar.gz` or `.txt` support bundle
  after reviewing it for secrets.
- Whether cloud security group and host firewall allow inbound UDP on
  `server.listen`.
- Redacted `server.listen` address/port if needed; keep the UDP port visible.
- `allowed_targets` summary with CIDR/domain and ports visible enough to debug.
- Prism version and Core tag if testing with Prism.
- Client-side error text or screenshots with secrets removed.

Do not send:

- `tgp.auth.psk`.
- Full private subscription URLs, proxy URLs, tokens, UUIDs, passwords, private
  keys, or API keys.
- SSH keys, account IDs, provider console screenshots with unrelated secrets, or
  full public/private infrastructure inventory.

Before posting publicly, review the output even though the diagnostics scripts
redact common secret forms.
